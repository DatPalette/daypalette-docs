# 阶段 1 低成本远端 JSON 方案

> 来源：原 `day_palette/docs/architecture/low-cost-remote-json-stage1-plan.md`，2026-04-20 提升到顶层共享层。
> 本文定义 DayPalette 在“内置基础库 + 远端 JSON 覆盖”阶段的共享发布策略、托管选型、版本边界与最小控制要求；客户端侧的请求、缓存、回退实现细节由 `day_palette` 单仓文档补充。

## 1. 文档目的

本文用于把 `BL-BR-10` 中“阶段 1：内置基础库 + 远端 JSON 覆盖”落成可执行技术方案，并明确前期低成本可落地的部署选择。

本文只讨论：

- 数据如何发布。
- 数据放在哪里。
- 哪种方案最适合 DayPalette 当前阶段。
- 最小的版本与控制边界应如何定义。

本文不讨论：

- HarmonyOS 客户端如何具体请求、缓存与回退。
- 完整 CMS 的权限与审核系统。
- 数据库建模。
- 后台管理台 UI。

---

## 2. 先说结论

对 DayPalette 当前阶段，最推荐的路径是：

1. **阶段 0**：先把本地硬编码数据迁到包内 `rawfile JSON`。
2. **阶段 1 验证版**：可先用 `GitHub Pages` 或免费静态托管验证“远端覆盖”是否值得做。
3. **阶段 1 正式低成本版**：优先使用**对象存储直出静态 JSON**。
4. **阶段 2**：只有当更新频率、协作人数、审核复杂度都明显上升后，再考虑 CMS。

换句话说，前期不建议为 DayPalette 先做“服务端系统”，而建议把远端更新理解为：

> **客户端读取一组受控的静态 JSON 文件，而不是调用一套复杂业务接口。**

---

## 3. 阶段 1 的目标形态

### 3.1 数据分层

阶段 1 的运行时数据来源分成三层：

1. **包内基础库**：`rawfile/palette-data/*.json`，始终随安装包存在。
2. **远端发布包**：托管在远端的静态 JSON 文件。
3. **本地缓存包**：客户端下载成功后保存到本地文件目录或 Preferences 指向的缓存信息。

### 3.2 运行原则

运行时遵守以下原则：

1. 包内基础库永远可用。
2. 远端包只负责覆盖内容，不负责替代客户端逻辑。
3. 远端拉取失败时，自动回退到包内基础库。
4. 客户端 UI 不感知数据来自包内还是远端，只读取统一 Repository 结果。

---

## 4. 推荐的数据发布结构

建议远端最少发布以下文件：

- `manifest.json`
- `palettes.v1.json`
- `collections.v1.json`
- `locale.zh-CN.v1.json`
- `locale.en-US.v1.json`

其中：

- `manifest.json` 负责告诉客户端当前可用的数据版本、文件地址、更新时间与校验信息。
- 其他 JSON 文件负责真正的内容。

建议 `manifest.json` 至少包含：

```json
{
  "schemaVersion": 1,
  "bundleVersion": "2026.04.07.1",
  "updatedAt": "2026-04-07T20:00:00+08:00",
  "files": {
    "palettes": "https://example.com/daypalette/palettes.v1.json",
    "collections": "https://example.com/daypalette/collections.v1.json",
    "localeZh": "https://example.com/daypalette/locale.zh-CN.v1.json",
    "localeEn": "https://example.com/daypalette/locale.en-US.v1.json"
  }
}
```

前期可以先不做复杂签名体系，但至少应保留：

- `schemaVersion`
- `bundleVersion`
- `updatedAt`
- 文件 URL
- 可选的 `hash` / `etag`

## 5. 四种低成本可落地方案对比

| 方案 | 典型载体 | 初始成本 | 开发复杂度 | 稳定性 | 发布控制 | 适合阶段 | 结论 |
|------|----------|----------|------------|--------|----------|----------|------|
| GitHub Pages | GitHub 仓库 + Pages 静态站点 | 很低 | 很低 | 中 | 中 | 最早期验证 | 适合快速验证远端覆盖是否值得做，不建议作为长期正式方案 |
| 对象存储 | 云厂商 OSS / OBS / COS / S3 类桶存储 | 低 | 低 | 高 | 高 | 阶段 1 正式低成本版 | **当前最推荐**；适合静态 JSON 直出、成本低、稳定、后续可扩展 |
| 静态托管 | Cloudflare Pages / Netlify / Vercel 等 | 很低到低 | 低 | 中到高 | 中 | 验证版或轻正式版 | 可作为 GitHub Pages 的替代，但发布控制和资源管理通常不如对象存储直接 |
| 云函数 | Serverless Function + 存储 | 中 | 中 | 中到高 | 高 | 需要额外逻辑时 | 前期不建议默认上；只有当需要签名、灰度、动态路由时才值得引入 |

---

## 6. 对比解读

### 6.1 GitHub Pages

优点：

- 几乎零成本。
- 上手最快。
- 改 JSON 文件即可发布。

缺点：

- 更适合验证，不太适合长期正式产品数据源。
- 缓存控制、版本控制与发布隔离能力有限。
- 资源管理方式偏工程化，不适合长期运营侧维护。

适合用途：

- 验证阶段 1 的价值。
- 验证客户端远端覆盖链路是否稳定。

### 6.2 对象存储

优点：

- 成本低，通常按存储量和流量计费，前期压力很小。
- 适合托管静态 JSON 文件。
- 稳定性和可控性通常优于纯免费静态站点。
- 后续可继续叠加 CDN、签名、版本化目录和灰度策略。

缺点：

- 需要云账号与基础配置。
- 比 GitHub Pages 多一层资源管理操作。

适合用途：

- DayPalette 当前阶段的正式低成本远端覆盖。

### 6.3 静态托管

优点：

- 部署体验好。
- 免费额度通常足够前期使用。
- 可以快速把 JSON 资源发布出去。

缺点：

- 长期资源治理与版本隔离未必比对象存储更清晰。
- 未来如果接入更细颗粒控制，迁移概率较高。

适合用途：

- 早期验证。
- 不想一开始碰对象存储后台时的过渡方案。

### 6.4 云函数

优点：

- 可以在返回 JSON 前做版本判断、签名校验、灰度策略、按客户端版本分流。
- 后续可自然演进到更复杂的数据分发体系。

缺点：

- 开发和维护复杂度明显上升。
- 已经不再是“最低成本、最低心智负担”的路线。

适合用途：

- 未来确实需要额外控制能力时，再引入。

## 7. 当前推荐方案

### 7.1 推荐路线

当前建议分两步：

1. **现在开始做阶段 0**：先把本地数据从 `LocaleData.ets` 硬编码迁到包内 `rawfile JSON`。
2. **准备进入阶段 1 时**：使用**对象存储直出静态 JSON** 作为正式方案。

若你希望在更低成本下先试通：

1. 先用 `GitHub Pages` 或免费静态托管验证一次。
2. 验证通过后，再迁到对象存储。

### 7.2 为什么不是一开始就上云函数或 CMS

因为 DayPalette 当前要解决的是：

- 偶尔更新 palette 与专题。
- 偶尔修正文案。
- 在不发版的情况下下线问题内容。

这类需求本质上仍属于**静态内容分发**，不是复杂业务系统。

---

## 8. 远端覆盖的最小控制要求

即使采用最低成本方案，也建议保留以下边界：

1. 远端数据只更新**内容**，不更新客户端逻辑。
2. 客户端必须校验 `schemaVersion`。
3. 客户端必须允许“远端失败 -> 自动回退包内基础库”。
4. 远端 JSON 应按版本号发布，避免直接覆盖无历史记录文件。
5. 若未来接入 Pro 内容，应确保免费与付费资源边界在客户端仍可校验。

---

## 9. 对 BL-BR-10 的落地建议

`BL-BR-10` 当前可拆成以下顺序：

1. 阶段 0：本地 `rawfile JSON` 文件化。
2. 阶段 0.5：增加本地读取层、mapper 和 fallback。
3. 阶段 1 文档化：确定 `manifest + 静态 JSON` 的远端结构。
4. 阶段 1 验证：用 GitHub Pages 或静态托管跑通一次远端覆盖链路。
5. 阶段 1 正式化：切到对象存储。

这样做的好处是：

- 每一步都可独立回滚。
- 不会在内容更新问题还没被证明成立前，就先付出服务端系统成本。
- 客户端数据结构先稳定，再决定是否升级远端分发方式。

---

## 10. 暂定结论

DayPalette 的阶段 1 不应理解为“开始做服务端”，而应理解为：

- **客户端继续掌握数据结构与逻辑边界。**
- **远端只托管静态 JSON 内容包。**
- **对象存储是最适合当前阶段的正式低成本方案。**
