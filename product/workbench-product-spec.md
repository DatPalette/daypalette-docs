## 2. 为什么现在需要它

DayPalette 当前面对的核心问题，已经不再只是“功能是否完整”，而是内容体量开始上升之后，如何高效地：

1. 批量查看几十到上百个配色盘。
2. 按场景持续扩充内容。
3. 用肉眼做筛选、比较、淘汰与保留。
4. 把人工判断沉淀成可复用的运营状态。

在这个阶段，真正的瓶颈不是复杂编辑表单，也不是多人审批流，而是：

1. 看得够不够快。
2. 比得够不够快。
3. 筛得够不够快。
4. 记不记得住为什么保留或淘汰。

因此，这个工具的正确定位不是传统后台，也不是桌面壳本身，而是一个“预览优先、策展优先、人工筛选优先”的配色策展工作台。

## 3. 产品目标与非目标

### 3.1 目标

MVP 阶段，这个工作台要解决以下问题：

1. 快速浏览一个场景下的大量配色盘。
2. 用接近真实产品界面的方式预览配色盘效果。
3. 快速筛选、标记、收藏、淘汰候选。
4. 对多个配色盘做并排比较。
5. 记录人工判断结果，避免完全依赖记忆。
6. 在最终入选后，把结果导出到正式资产链路。

### 3.2 非目标

MVP 阶段暂时不做：

1. 多人协作与权限系统。
2. 复杂审批流。
3. 在线 CMS。
4. 运行时动态下发到 App。
5. 完整通用代码编辑器平台。
6. 完整 Git 图形客户端。
7. 复杂插件系统。

### 3.3 能力优先级

建议按以下优先级理解能力建设顺序：

1. 预览
2. 筛选
3. 比较
4. 标记与状态沉淀
5. 基础编辑
6. 导出与发布

如果优先级反过来，很容易做成一个能录入数据、却不适合策展的后台。

## 4. 全局页面架构

### 4.1 整体布局建议

建议采用典型工作台布局：

```text
+----------------------------------------------------------------------------------+
| Top Bar: 场景切换 | 搜索 | 结果数 | 视图切换 | 导入 | 校验 | 导出 |
+----------------------------------------------------------------------------------+
| Left Filter Rail   | Main Workspace                                             |
| 场景               |                                                            |
| 状态               |                                                            |
| 风格               |                                                            |
| 情绪               |                                                            |
| 安全等级           |                                                            |
| 评分               |                                                            |
| 是否入选           |                                                            |
+----------------------------------------------------------------------------------+
| Bottom Utility Bar: 已选数量 | 批量标记 | 对比 | 恢复 | 备注 | 发布前检查 |
+----------------------------------------------------------------------------------+
```

### 4.2 设计原则

1. 左侧永远保留筛选面板。
2. 顶部永远保留场景切换、搜索和视图切换。
3. 底部或浮动工具栏承担批量操作。
4. 详情不应强制跳页，可优先用右侧抽屉或浮层展开。

这能最大化维持“看大量候选时的连续性”。

## 5. 核心工作区

### 5.1 配色盘总览页

目标：

1. 快速进入某个场景。
2. 浏览候选总量。
3. 进行初步筛选。
4. 快速把候选送入预览或对比流程。

必须支持的快捷操作：

1. 单击进入详情。
2. 勾选加入批量操作。
3. 快捷标记 `shortlisted`。
4. 快捷标记 `rejected`。
5. 快捷加入对比。

总览页应允许直接完成第一轮筛选，不必频繁进入详情。

### 5.2 批量预览页

这是整个工作台的核心页面，目标是：

1. 让你在短时间内扫过大量候选。
2. 直接看到配色盘进入产品后的大致效果。
3. 快速做“保留 / 待定 / 淘汰”的审美判断。

每张预览卡建议分成四层：

1. 第一层：模拟真实页面的视觉缩略图。
2. 第二层：配色盘名称、场景标签、状态。
3. 第三层：评分、风险提示、是否入选。
4. 第四层：快捷操作按钮。

### 5.3 对比页

用于解决“相似候选之间谁更稳、谁更值得保留”的问题。

推荐对比维度：

1. 页面整体气质。
2. 层次感是否足够。
3. 标题与标签可读性。
4. 与同组候选的差异度。
5. 是否更符合当前场景语义。

必须支持：

1. 一键保留一个并批量淘汰其它。
2. 将对比结论直接写入 `reviewNotes`。
3. 重新加入待定池。
4. 自动推荐相似配色盘进入对比池。

### 5.4 入选与淘汰管理页

这个页面不是为了再次预览，而是为了沉淀和追溯决策。

需要支持：

1. 按场景查看最终候选集。
2. 一键恢复误淘汰项。
3. 从已入选集生成发布候选集。
4. 查看历史备注与最近操作。

### 5.5 单个配色盘详情页

详情页承担两个职责：

1. 做更精细的人工判断。
2. 做少量必要编辑。

它不应成为高频主入口，更适合承接：

1. 更细的预览切换。
2. 评分与备注补录。
3. 原始数据查看入口。
4. 相似候选推荐。

### 5.6 发布前检查页

用于把“策展工作流”衔接到“正式发布工作流”。

需要支持：

1. 区分阻塞错误与非阻塞警告。
2. 从错误直接跳转回对应配色盘。
3. 只导出当前场景或当前候选集。
4. 先构建预览 bundle，再决定是否正式发布。

## 6. 预览系统原则

### 6.1 预览不是简单色块

如果工作台只展示三块颜色，它的价值会很有限。你真正需要看到的是：

1. 这个配色盘进入产品首页后是什么感觉。
2. 标题与副标题压在这个配色上是否舒服。
3. 色块之间是否有层次。
4. 页面整体气质是否成立。

所以预览必须尽量接近真实产品界面，而不是停留在资产表格层。

### 6.2 建议至少提供 4 类预览模板

1. 今日页主卡缩略图。
2. 列表卡片缩略图。
3. 详情页氛围图。
4. 对比模式。

### 6.3 为什么不应该继续依赖模拟器

模拟器适合：

1. 功能验证。
2. 细节验收。
3. 单个页面效果确认。

模拟器不适合：

1. 大量候选并排比较。
2. 快速场景切换。
3. 批量淘汰。
4. 内容策展工作流。

因此模拟器在这个阶段应降级为“最终验收工具”，而不是“主要预览工具”。

## 7. 字段模型

建议把工作台里需要的字段分成三层理解。

### 7.1 复用现有资产语义字段

这些字段来自 palette 本身，描述“它是什么”和“它适合什么”：

- `id`
- `slug`
- `nameZh`
- `nameEn`
- `primaryColorId`
- `secondaryColorId`
- `accentColorId`
- `occasionId`
- `styleTags`
- `moodTags`
- `seasonTags`
- `safetyLevel`

### 7.2 工作台运营字段

这些字段描述“它目前被如何看待”。

| 字段 | 类型 | 作用 |
|------|------|------|
| `reviewStatus` | enum | 描述候选当前处于 `candidate / shortlisted / approved / rejected / archived` 哪个阶段。 |
| `curationScore` | number | 记录人工主观评分，用于排序与初筛。 |
| `isSelected` | boolean | 表示是否进入当前正式入选集。 |
| `reviewNotes` | string | 记录人工判断结论。 |
| `rejectionReason` | string \| null | 记录淘汰原因，减少重复比较。 |
| `lastReviewedAt` | string | 记录最近一次人工判断时间。 |

### 7.3 发布辅助字段

这些字段描述“它是否已进入发布准备”。

| 字段 | 类型 | 作用 |
|------|------|------|
| `readyForBundle` | boolean | 标记是否可进入 bundle 候选。 |
| `previewVariant` | enum | 指定默认预览模板，例如 `today-hero / list-card / detail-atmosphere`。 |
| `validationFlags` | string[] | 记录当前存在的校验提示或人工关注项。 |

当前最值得优先稳定的，是：

- `reviewStatus`
- `curationScore`
- `rejectionReason`
- `reviewNotes`
- `isSelected`
- `readyForBundle`

## 8. 状态机与约束

### 8.1 最小状态机

建议先跑通这条最小状态机：

```text
candidate -> shortlisted -> approved
candidate -> rejected
shortlisted -> rejected
approved -> shortlisted
rejected -> shortlisted
rejected -> archived
approved -> archived
```

这已经足够表达：

1. 初筛。
2. 中筛。
3. 入选。
4. 淘汰。
5. 归档。
6. 误操作恢复。

### 8.2 关键流转规则

1. `candidate -> shortlisted`
   - 说明值得进一步比较。
   - 需要记录 `lastReviewedAt`，可选记录评分与备注。

2. `candidate / shortlisted -> rejected`
   - 直接或经对比后淘汰。
   - `rejectionReason` 必填。

3. `shortlisted -> approved`
   - 已完成精筛或对比，确认进入可用候选集。
   - 建议同步更新 `curationScore`、`reviewNotes`，并可把 `isSelected` 设为 `true`。

4. `approved -> shortlisted`
   - 后续新增更优候选，需要重新比较。

5. `rejected -> shortlisted`
   - 人工复核后恢复，防止误淘汰成为不可逆动作。

6. `approved / rejected -> archived`
   - 当前运营周期结束，或资产不再参与后续比较。

### 8.3 字段约束关系

1. 当 `reviewStatus = rejected` 时：
   - `rejectionReason` 应为必填。
   - `isSelected` 必须为 `false`。

2. 当 `reviewStatus = approved` 时：
   - `curationScore` 建议必填。
   - `reviewNotes` 建议至少有简短说明。
   - `isSelected` 可为 `true` 或 `false`，两者不应等同。

3. 当 `reviewStatus = archived` 时：
   - 不应再默认出现在主工作区。
   - 只能在历史视图中查看。

### 8.4 批量操作规则

1. 批量标记为 `shortlisted`：适合第一轮保留。
2. 批量标记为 `rejected`：适合明显不符合场景的候选。
   - 应支持统一填写默认 `rejectionReason`。
3. 批量加入入选集：
   - 仅允许 `approved` 状态的项批量加入。

## 9. 推荐的数据链路

建议链路如下：

1. 资产真相源保存在独立资产仓库。
2. 工作台读取 `source` 数据与策展状态。
3. 工作台主要负责预览、筛选、标记和基础编辑。
4. 发布前运行校验。
5. 构建移动端消费的 bundle。
6. 由脚本同步到移动端工程。

共享流程细节请优先参考 `daypalette-docs`，这里不再重复写多仓协作规则。

---

## 10. 分阶段实现建议

### 10.1 阶段一：只做看、筛、标

先实现：

1. 总览页。
2. 批量预览页。
3. 对比页。
4. 基础筛选。
5. 状态标记。
6. 备注记录。

这一阶段的目标只有一个：

让你能高效处理几十到上百个候选配色盘。

### 10.2 阶段二：补基础编辑

再实现：

1. 详情页。
2. 基础字段编辑。
3. 原始 JSON 编辑入口。
4. 批量标签修改。

### 10.3 阶段三：补发布闭环

最后实现：

1. 校验面板。
2. bundle 构建。
3. 提交到资产仓库或导出本地。
4. 与移动端同步脚本衔接。

---

## 11. 成功标准

这套工作台是否成功，不看它是否像一个成熟后台，而看下面几个指标：

1. 你能否在一个页面内快速扫完 50 到 100 个配色盘。
2. 你能否在 10 分钟内完成一次场景内的初筛。
3. 你能否很快回忆起为什么某个候选被淘汰。
4. 你能否在不打开模拟器的前提下完成大部分内容判断。
5. 你能否把最终入选结果顺畅推进到移动端资源链路。

如果这几件事做到了，这个工作台就是成功的。
