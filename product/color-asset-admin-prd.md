## 1. 文档定位

本文用于定义 DayPalette 新一轮“配色资产管理后台”的共享层产品判断。

本文重点回答：

1. 新后台到底要解决什么问题。
2. 后台首页信息架构应该如何组织。
3. “基础数据”与“集合数据”分别包含什么对象。
4. 首版必须具备哪些 CRUD、校验与写回能力。
5. 哪些需求当前明确不做。

本文面向：

1. DayPalette 主理人。
2. 后台 monorepo 实现者。
3. 需要理解新运营边界的 AI Agent。

---

## 2. 背景

当前 DayPalette 的配色资产仍然保存在 `day_palette` 工程中，主要文件为：

1. `entry/src/main/resources/rawfile/palette-data/base-colors.v1.json`
2. `entry/src/main/resources/rawfile/palette-data/palettes.v1.json`
3. `entry/src/main/resources/rawfile/palette-data/collections.v1.json`

当前痛点不是“没有 JSON”，而是：

1. 日常维护仍然偏工程师直接改文件。
2. 各文件里的可枚举字段散落在记录内部，无法单独管理。
3. 新增、修改、删除资产时，引用关系和字段口径容易失控。
4. 旧的多仓运营方案已经归档，必须基于新的共享层文档重新定义边界。

---

## 3. 一句话结论

DayPalette 新一轮配色运营首版应采用一个**本地优先、单人使用、左侧菜单右侧内容区的管理后台**，由“基础数据管理”与“集合数据管理”两大菜单构成，并围绕 `base-colors.v1.json`、`palettes.v1.json`、`collections.v1.json` 以及其可枚举字段提供完整的新增、修改、软删除和校验能力。

---

## 4. 产品定义

### 4.1 它是什么

它应被定义为：

1. 一个服务于单人本地运营的轻量后台。
2. 一个围绕现有 JSON 资产进行维护的管理工具，而不是重新发明内容系统。
3. 一个先解决“结构化维护效率”的工具，而不是先追求复杂工作流的系统。
4. 一个由前端界面和本地服务端共同构成的 monorepo 产品。

### 4.2 它不是什么

当前阶段，它不是：

1. 远端在线 CMS。
2. 多人协作审批平台。
3. 数据库优先的内容中台。
4. 直接面向 App 终端用户的运行时服务。
5. 追求复杂可视化排版的设计工具。

---

## 5. 目标用户与核心任务

### 5.1 当前目标用户

当前只服务一类用户：

1. 负责配色资产维护的单人运营者 / 主理人。

### 5.2 核心任务

这个用户每天真正要做的事情是：

1. 管理配色资产里反复出现的基础口径，例如场合、风格、季节、安全等级。
2. 维护基础色数据，包括颜色名称、色值、标签、状态。
3. 维护配色盘数据，包括三色组合、场合、风格、季节与来源信息。
4. 维护合集数据，包括主题、成员、封面、发布模式与状态。
5. 在修改过程中及时发现引用断裂、枚举值漂移和非法删除。
6. 把修改结果稳定写回到 DayPalette 当前保存原始文件的位置。

## 6. 首版范围

### 6.1 顶层信息架构

后台采用固定壳层：

1. 左侧：菜单导航。
2. 右侧：列表、过滤器、详情编辑区。

顶层只分两大类菜单：

1. 基础数据
2. 集合数据

### 6.2 基础数据

“基础数据”指三个主数据文件中所有可枚举、可复用、应单独维护的字段集合。

首版至少覆盖以下口径：

| 菜单 | 主要作用 | 典型来源字段 |
| --- | --- | --- |
| 场合管理 | 维护场合类单选 / 多选口径 | `occasionId`、`occasionTags` |
| 风格标签管理 | 维护风格标签集合 | `styleTags` |
| 季节标签管理 | 维护季节标签集合 | `seasonTags` |
| 心情标签管理 | 维护情绪标签集合 | `moodTags` |
| 色系管理 | 维护基础色色系 | `colorFamily` |
| 冷暖管理 | 维护冷暖属性 | `tone` |
| 明度等级管理 | 维护明度枚举 | `lightnessLevel` |
| 饱和度等级管理 | 维护饱和度枚举 | `saturationLevel` |
| 安全等级管理 | 维护配色安全等级 | `safetyLevel` |
| 集合主题管理 | 维护合集主题类型 | `themeType` |
| 发布模式管理 | 维护合集发布模式 | `releaseMode` |
| 来源类型管理 | 维护配色来源类型 | `sourceType` |
| 状态管理 | 维护各对象状态口径 | `status` |

约束：

1. 单选与多选字段都算“基础数据”。
2. 布尔字段如 `isPro`、`fitPhotoScenario`、`isNeutralCore` 不进入基础数据菜单，仍在具体记录表单里直接编辑。
3. 基础数据必须支持单独新增、修改、停用和删除校验。

### 6.3 集合数据

“集合数据”指当前直接被 App 消费的三份主数据：

1. Base Colors
2. Palettes
3. Collections

每种数据都必须支持：

1. 列表查看。
2. 搜索与筛选。
3. 新增。
4. 修改。
5. 软删除。
6. 软删除前引用检查。

## 7. 页面要求

### 7.1 基础数据页面

基础数据页面首版采用“表格 + 右侧抽屉 / 弹层表单”的轻量模式。

每个基础数据页面至少展示：

1. 选项 ID。
2. 中文名。
3. 英文名。
4. 排序值。
5. 是否启用。
6. 被引用次数。

每个基础数据页面至少支持：

1. 新增选项。
2. 编辑名称、说明、排序。
3. 停用未准备继续使用的选项。
4. 在通过引用检查后执行软删除。
5. 查看当前被哪些数据记录引用。

### 7.2 Base Colors 页面

Base Colors 页面首版采用“网格 + 卡片”模式。

每张卡片至少展示：

1. 色块。
2. 颜色名称。
3. `hex`。
4. `tone`、`lightnessLevel`、`saturationLevel`。
5. `styleTags`、`occasionTags`、`seasonTags`。
6. 状态。

编辑页至少支持：

1. 直接输入 / 修改 `hex`。
2. 维护中英文名称。
3. 选择基础数据项。
4. 编辑布尔字段。
5. 软删除前检查是否被 palette 引用。

### 7.3 Palettes 页面

Palettes 页面首版也采用“网格 + 卡片”模式，但重点展示三色关系。

每张卡片至少展示：

1. 三个色块预览。
2. `id` 与 `slug`。
3. 场合、风格、季节、心情、安全等级。
4. `isPro`、`status`。

编辑页至少支持：

1. 通过选择器引用已有 Base Color。
2. 编辑单选 / 多选基础数据字段。
3. 编辑 `sourceCollectionIds`。
4. 软删除前检查是否被 collection 引用。

### 7.4 Collections 页面

Collections 页面首版采用“列表为主，卡片为辅”的模式。

每条记录至少展示：

1. 集合名称。
2. `themeType`。
3. 封面配色。
4. 成员数量。
5. `isPro`、`releaseMode`、`status`。

编辑页至少支持：

1. 编辑中英文名称与描述。
2. 选择封面配色。
3. 管理 `paletteIds` 顺序。
4. 维护场合 / 风格标签。
5. 软删除前检查是否仍引用有效 palette。

## 8. 通用交互要求

### 8.1 首版界面原则

首版界面只需要高效、清楚，不需要复杂炫技。

要求：

1. 整体设计必须以 [`../design/shared-design-language.md`](../design/shared-design-language.md) 为真相源，而不是临时自行发挥。
2. 后台仍遵守共享设计语言里的纸质画廊感、编辑级排版、克制材质和语义令牌基线。
3. 左侧导航稳定固定，右侧内容页按资源类型切换。
4. 颜色与配色数据优先使用卡片和网格展示。
5. 表单优先使用抽屉或独立详情区，不在列表内直接改大段字段。

### 8.2 通用操作

所有资源页都应具备：

1. 搜索。
2. 基础筛选。
3. 最近修改后即时刷新。
4. 保存成功 / 失败反馈。
5. 删除确认。

### 8.3 删除规则

删除不是无脑执行。

要求：

1. 所有删除动作默认都是软删除，不物理移除原始记录。
2. 被引用的基础数据不能执行软删除。
3. 被 palette 引用的 Base Color 不能执行软删除。
4. 被 collection 引用的 Palette 不能执行软删除。
5. 删除失败时要直接告诉用户引用来源。

### 8.4 已定数据合同

以下口径在当前阶段已经定稿，不再在实现阶段临时改写：

1. `base-colors.v1.json`、`palettes.v1.json`、`collections.v1.json` 的软删除统一写成 `status=deleted`。
2. 主数据软删除时，必须同时保留 `previousStatus` 与 `deletedAt`。
3. 字典项用 `isActive=false` 表示停用，用 `isDeleted=true` 表示软删除；两者不能混为一谈。
4. 详细字段定义与 `dictionaries.v1.json` 草案结构见 [`../architecture/color-asset-admin-data-contract.md`](../architecture/color-asset-admin-data-contract.md)。

---

## 9. 非目标

首版明确不做：

1. 用户系统、权限系统、角色系统。
2. 云端数据库与线上托管。
3. 复杂审批流、评论流、任务流。
4. App 运行时动态下发。
5. 富文本编辑器与复杂可视化编排。
6. 通用 JSON 编辑器模式作为主入口。

---

## 10. 成功标准

如果首版成功，应满足以下结果：

1. 运营者不需要手工打开 JSON，也能完成三类主数据的新增、修改、软删除。
2. 所有枚举型字段都能通过单独页面维护，而不是靠人工记忆字符串。
3. 错误引用、非法软删除、字段漂移能在保存前被拦住。
4. 修改结果能稳定写回 DayPalette 当前原始文件目录。
5. 整个后台仍保持轻量，不引入数据库和多人协作复杂度。

---

## 11. MVP 验收口径

MVP 至少满足：

1. 后台能启动并读取 `day_palette` 当前三份主数据文件。
2. 左侧菜单清晰分为“基础数据”和“集合数据”两组。
3. 基础数据至少能完整维护本文 §6.2 列出的口径。
4. Base Colors、Palettes、Collections 三页都支持新增、修改、软删除。
5. 对任意对象执行软删除前，系统都会先做引用检查并在失败时展示来源。
6. 调整基础数据后，相关表单选项会即时更新。
7. 颜色与配色展示采用网格 / 卡片，满足基础可视化判断。
8. 所有保存动作都能把结果写回文件，并保留结构化合法性。

---

## 12. 后续议题

本文不展开但后续大概率要补的内容包括：

1. 是否在首轮迁移时一次性补齐所有字典项的中英说明、别名与描述文案。
2. 是否补充简单的发布检查页或变更摘要页。
3. 是否需要把 locale 维护单独抽成第三类菜单。
4. 是否需要在第二阶段补充导入 / 导出、批量编辑与 Git 辅助能力。
